# Сервис отправки заявок

Небольшой backend на Node.js 20 принимает заявку калькулятора и отправляет её в Telegram и MAX. Он возвращает `success: true` только после подтверждённого ответа обоих API. В коде нет токенов, ID адресатов и телефонных номеров.

## Запуск

Сервис не читает `.env` сам: переменные должен передать процесс-менеджер, контейнер или shell. В каталоге сервиса выполните:

```bash
npm start
```

Проверка после запуска:

```bash
curl http://127.0.0.1:8787/health
```

Ответ: `{"status":"ok"}`.

## Переменные окружения

Обязательные переменные:

| Переменная | Назначение |
| --- | --- |
| `ALLOWED_ORIGIN` | Единственный origin сайта, например `https://example.ru`. Путь и query запрещены. |
| `TELEGRAM_BOT_TOKEN` | Токен Telegram-бота. |
| `TELEGRAM_CHAT_ID` | ID чата или пользователя Telegram, куда бот отправляет заявки. |
| `MAX_BOT_TOKEN` | Access token MAX. Сервис передаёт его как `Authorization: <token>`, без query-параметра. |
| `MAX_USER_ID` или `MAX_CHAT_ID` | Ровно один int64 ID адресата MAX. |
| `MAX_CA_PEM` | PEM-сертификат Минцифры. Можно передать многострочный PEM или строку с литералами `\n`. |

Опциональные переменные:

| Переменная | По умолчанию | Назначение |
| --- | ---: | --- |
| `HOST` | `0.0.0.0` | Адрес прослушивания. |
| `PORT` | `8787` | Порт HTTP-сервера. |
| `UPSTREAM_TIMEOUT_MS` | `8000` | Таймаут одного запроса к Telegram или MAX, 1000–30000 мс. |
| `RATE_LIMIT_MAX` | `20` | Число `POST /orders` от одного клиента за окно. |
| `RATE_LIMIT_WINDOW_MS` | `60000` | Размер окна локального rate limit. |
| `STATE_FILE` | не задан | Файл для сохранения идемпотентности между перезапусками. |
| `STATE_MAX_ENTRIES` | `10000` | Максимум действующих записей в состоянии. При заполнении сервис отвечает 503. |
| `STATE_TTL_MS` | `604800000` | Срок хранения состояния заявки: 7 дней. Допустимый диапазон — от 1 минуты до 365 дней. |
| `TRUST_PROXY` | `false` | Разрешает брать первый IP из `X-Forwarded-For`. Включайте только за доверенным reverse proxy, который перезаписывает этот заголовок. |

Шаблон без значений находится в `.env.example`. Не добавляйте рабочий `.env` в Git.

### Важное ограничение MAX

Номер телефона **не является** `MAX_USER_ID` и не подходит для `MAX_USER_ID` или `MAX_CHAT_ID`. Адресат должен сначала запустить бота. Бот должен пройти модерацию MAX. После события `bot_started` возьмите настоящий `user_id` из webhook или Long Polling и задайте его в `MAX_USER_ID`. Для группового чата используйте `chat_id` из события MAX.

Сервис отправляет MAX-запрос на официальный адрес `POST https://platform-api2.max.ru/messages` с `user_id` либо `chat_id` в query и токеном в заголовке `Authorization`. Согласно [документации MAX](https://dev.max.ru/docs-api/methods/POST/messages), для нового домена нужен сертификат Минцифры. Сервис добавляет `MAX_CA_PEM` к `tls.rootCertificates`, сохраняя стандартные корневые сертификаты Node.js.

## HTTP API

### `POST /orders`

Сервис принимает запрос только с заголовком `Origin`, который точно совпадает с `ALLOWED_ORIGIN`. Он поддерживает `application/json` и `text/plain` с JSON-строкой — второй формат уже использует калькулятор. Максимальный размер тела — 200 КиБ. `plainText` ограничен 4000 символами по лимиту MAX.

Обязательны текущие поля payload калькулятора: `schemaVersion: 2`, `id`, `createdAt`, сведения о лодке и размерах, `customer`, подтверждение размеров, `plainText` и `idempotencyKey`. Поле `site.origin` должно точно совпадать с `ALLOWED_ORIGIN`. Значение заголовка `Idempotency-Key`, если он передан, должно совпадать с полем тела.

Успешный ответ имеет HTTP 200:

```json
{
  "success": true,
  "orderNumber": "BM-...",
  "channels": {
    "telegram": true,
    "max": true
  }
}
```

Если один канал не подтвердил доставку, сервис отвечает HTTP 502 и `success: false`. Повторный запрос с тем же `idempotencyKey` и тем же содержимым отправляется только в провалившийся канал. Поле `createdAt` можно обновить при повторе; изменение данных заявки с тем же ключом даёт HTTP 409.

Основные коды ошибок: 400 — неверное тело, 403 — запрещённый origin, 413 — тело больше лимита, 415 — неподдерживаемый Content-Type, 429 — локальный rate limit, 502 — неполная доставка, 503 — ошибка состояния или сервиса.

### `OPTIONS /orders`

Preflight разрешён только для точного значения `ALLOWED_ORIGIN`. Сервис объявляет методы `POST, OPTIONS` и заголовки `Content-Type, Idempotency-Key`.

### `GET /health`

Проверяет, что HTTP-процесс работает. Endpoint не раскрывает конфигурацию и не проверяет внешние API.

## Идемпотентность и хранение

Без `STATE_FILE` состояние хранится в памяти и теряется при перезапуске. С `STATE_FILE` сервис сохраняет состояние через временный файл и атомарный `rename`. Файл содержит ключ, SHA-256 отпечаток заявки, номер заказа, статусы каналов и время; текст заявки и контакты клиента туда не записываются.

Для production `STATE_FILE` нужно размещать на постоянном томе. Без него перезапуск между частичной доставкой и повтором запроса может привести к повторному сообщению в уже сработавший канал.

Сервис удаляет записи через `STATE_TTL_MS` после их последнего обновления. Он очищает просроченные записи при загрузке файла и перед чтением или созданием записи. Поэтому просроченная завершённая заявка освобождает место при достижении `STATE_MAX_ENTRIES`; действующие записи сервис не вытесняет, чтобы не нарушить идемпотентность. Повтор после TTL считается новой доставкой и может повторно отправить заявку в оба канала.

Один локальный файл подходит только одному процессу. Несколько реплик потребуют общего хранилища с распределённой блокировкой. Локальный rate limit также действует отдельно в каждом процессе.

Проверка `Origin` защищает от отправки с посторонней веб-страницы, но не является авторизацией: HTTP-клиент может подделать этот заголовок. Перед публичным запуском поставьте сервис за reverse proxy/WAF, ограничьте частоту запросов на его уровне и при заметном спаме добавьте CAPTCHA с серверной проверкой.

## Docker

Из корня репозитория:

```bash
docker build -t eva-order-service integrations/order-service
docker volume create eva-order-state
docker run --rm --env-file integrations/order-service/.env \
  --mount type=volume,source=eva-order-state,target=/data \
  -e STATE_FILE=/data/orders-state.json \
  -p 8787:8787 eva-order-service
```

Постоянный том должен быть доступен пользователю контейнера `node`. Если платформа монтирует каталог с другими правами, заранее выдайте ему право записи.

## Тесты

Из каталога `integrations/order-service`:

```bash
npm test
npm run check
```

Тесты используют локальный HTTP-сервер и моки Telegram/MAX. Сетевые запросы к мессенджерам не выполняются.
